# plan.md — этапы разработки UNLHA

Каждый этап несёт законченную функциональность и проверяем в эмуляторе MAME
(через `make img`) или на реальном Sprinter. Этапы идут по возрастанию
сложности; SRAM-кэш подключается, когда базовая распаковка уже корректна.

Легенда статуса: [ ] не начато · [~] в работе · [x] готово.

---

- [x] Битовый поток MSB-first (модель LHA, побитовое заполнение).

## Этап 4 — Декодер -lh1- (LZHUF), эталонная реализация в DRAM  [x]
**Цель:** второй распространённый метод (присутствует в тестах).
- [x] Адаптивный Хаффман: построение (`StartHuff`), перестройка (`Lh1Reconst`),
      инкремент частот и ребаланс (`Lh1Update`/`Lh1UpdateSwap`).
- [x] Декодирование расстояния (`DecodePosition`: 8-битный префикс через `d_code`/
      `d_len` в `dtables.inc` + явные младшие биты), окно 4 KB, длина 3..60.
- [x] Интеграция в общий конвейер вывода и CRC16 (`DecodeLh1`, буфер 4 КБ в WIN3,
      окно в WIN2; `IsLh1`/`MethodLh1` в `ExtractEntry`). Сборка чистая.
- [x] Проверка на железе: `CPM431.lzh`, `GS19.LZH`, `WINPROFI.LZH` → CRC `OK`
      (подтверждено пользователем 2026-06-15).
- [x] Выходной каталог приводится к абсолютному пути (`MakePathAbsolute`/`GetCurDir`,
      по образцу sprinter-unzip) — устранена путаница относительных путей DSS.
**Найденные баги транскрипции (исправлены):** (1) `DecodeChar` возвращал символ через
`Lh1Cv`, который `Lh1Update` использует как рабочий счётчик и обнуляет → весь вывод
из нулей; (2) `Lh1Reconst` LDDR-сдвиг указывал на младший байт слова вместо старшего
(копирование сверху вниз) — проявлялось на файлах >32K символов. **MVP закрыт:
-lh0-, -lh5-, -lh1-.**

## Этап 5 — Перенос горячего кода в SRAM CACHE (ускорение)  [~]
**Цель:** максимальная скорость без изменения функциональности. Идём
инкрементально (как sprinter-unzip: сначала CRC как «канарейка» кэша).
- [x] Шаг 5A: инфраструктура `src/cache.asm` — `EnterCacheWindow`/
      `RestoreSystemWindow` (порты #FB/#7B/#3C/#1FFD), `InitSramBundle` (копия
      бандла в SRAM + генерация таблицы CRC16), DISP-бандл, `ASSERT` раскладки.
- [x] Шаг 5A: CRC16 переведён на таблицу (256 слов, lo/hi постранично в SRAM
      #3800/#3900) и исполняется из SRAM (`CacheCrc16Update` #3A00). Алгоритм
      сверен host-side: совпадает с побитовым CRC16/ARC. **Подтверждено на
      железе 2026-06-15 (CRC OK, кэш-порты работают).**
- [x] Шаг 5B: массивы дерева -lh1- (TextBuf/freq/son/prnt ≈ 8.5K) перенесены в
      SRAM WIN0 (#0000/#1000/#1500/#1A00); декод держит CASH_ON весь проход,
      DSS только на границах (`RefillInBuf` чтения, `Lh1Flush` записи) —
      трамплинятся по флагу `CacheHeld` (общий битридер с lh5 не затронут).
      Код декодера пока в DRAM (WIN1) — исполняется при CASH_ON без ловушки
      «возврат в SRAM». **Подтверждено на железе 2026-06-15 (CRC OK).**
      lh5 не трогаем: его таблицы (кольцо 8K + CTable 8K) в 16K SRAM не влезают.
- [x] Шаг 5C: вычислительное ядро -lh1- (DecodeChar/DecodePosition/Lh1Update/
      Lh1UpdateSwap/Lh1Reconst/Lh1GetWord/Lh1PutWord) вынесено в SRAM-бандл
      (DISP #2200, 901 байт). Ядро не делает DSS — нет ловушки «возврат в SRAM».
      Out-вызовы только в DRAM `GetBits` (сам ре-входит в кэш перед RET через
      `RefillInBuf`-трамплин). Главный цикл, match-copy, `Lh1PutByte`,
      `Lh1Flush`, `StartHuff` остаются в DRAM. Сборка чистая, ASSERT раскладки
      прошли, адреса разрешились (ядро в SRAM, I/O в WIN1).
      ИСПРАВЛЕНИЕ после первого теста (2 CRC ERROR на WINUSER/WINWINDS.DOC):
      модель прерываний выровнена строго по sprinter-unzip (правильный референс
      по DSS-границам кэша). Было ошибочно взято из gifview: `RestoreSystemWindow`
      делал `EI`, поэтому DSS на границах шёл с прерываниями, и прерывание при
      переключении карты/кэша портило состояние (тайминг-зависимо → проявилось
      только с кодом в SRAM). Стало: `RestoreSystemWindow` не трогает прерывания,
      `DI` держится весь декод (Restore->DSS->Enter), `EI` — явно на трёх
      границах верхнего уровня (InitSramBundle, конец DecodeLh1, Crc16Update вне
      кэша). **Ждёт повторной проверки на железе.**
**Проверка:** результаты идентичны этапам 3–4 (CRC/байты), измеримое ускорение
(сравнить время на одних и тех же файлах до/после).

## Этап 5O — Оптимизация горячего цикла (ускорение поверх SRAM)  [ ]
**Контекст:** базовый перенос в SRAM (5A–5C) сделан, но решение медленнее unzip.
Причины: (а) `-lh1-`/LZHUF алгоритмически тяжелее DEFLATE (адаптивный Хаффман
перестраивает дерево на каждый символ + обход по 1 биту/уровень); (б) «наивный»
порт LZHUF.C — горячий цикл полон `CALL`-оверхеда и DRAM-доступов. Цель — сократить
оверхед; функциональность и CRC не меняются. Идём по убыванию отдачи.
- [x] **5O-1. Инлайн word-доступов + дешёвые адреса.** Раунд 1: `son[R]`/`freq[R]`
      через константный `LD HL,(nn)`; инлайн word-доступа в `DecodeChar.walk` и
      `Lh1Update`; `freq[c]++` на месте + `freq[l]=freq[c+1]` по смежному адресу.
      Раунд 2: инлайн всех `Lh1GetWord`/`Lh1PutWord` в `Lh1UpdateSwap`.
      **Подтверждено на железе 2026-06-16: CRC OK, заметно быстрее.**
- [~] **5O-2. Ускорение чтения бит.** Раунд 1 (сделано): `GetBits` извлекает верхние
      B бит сдвигом ВЛЕВО за B итераций вместо 16-B вправо — для частого B=1 (обход
      дерева) 1 итерация вместо 15, убран тяжёлый 15-кратный сдвиг. Общий с lh5,
      семантика идентична. Сборка чистая. **Ждёт железа** (CRC=OK + замер).
      Раунд 2 (сделано): SRAM-копии `CacheGetBits`/`CacheFillBuf`/`CacheInByte` в
      бандле lh1 — `DecodeChar`/`DecodePosition` больше не платят DRAM-выборку кода
      на каждый бит. Состояние битридера общее в WIN1; дозагрузка входа -> DRAM
      `RefillInBuf` (трамплин, ре-вход в кэш перед RET). lh5 продолжает звать
      DRAM-оригиналы. Тела байт-в-байт копии, сборка чистая. **Ждёт железа.**
- [~] **5O-3. Переменные lh1 в SRAM.** Сделано: блок `Lh1Vars` (#3000-#3029, 22
      переменные) перенесён из DRAM в SRAM WIN0 — горячий цикл (Lh1Cv/Lh1K/Lh1L и
      др.) без wait. Init `Lh1OutPos` перенесён после Enter (SRAM не загружается
      лоадером; все переменные пишутся до чтения при CASH_ON). ASSERT раскладки.
      БАГ И ФИКС: `Lh1Flush` перечитывал число байт из `(Lh1OutPos)` ПОСЛЕ
      `RestoreSystemWindow` (кэш выключен для DSS-записи) → при CASH_OFF #3002 это
      НЕ SRAM, читался мусор/0 → писалось 0 байт (файлы *.doc нулевой длины, при
      этом CRC=OK, т.к. счётчик CRC читался ДО выключения). Исправлено: счётчик
      сохраняется в регистре (PUSH BC/POP DE) до Restore. ПРАВИЛО: в трамплине
      Restore->DSS->Enter нельзя читать/писать переменные из SRAM WIN0.
      **Ждёт железа** (CRC=OK + ненулевые файлы). Осталось (опц.): `dtables` в SRAM.
- [~] **5O-4. Ускорить `-lh5-`.** Замечание: кольцо нельзя держать в SRAM (его читает
      `Dss.Write` при CASH_OFF), CTable+кольцо в 16K SRAM не влезают — полный перенос
      в SRAM малоэффективен. Поэтому идём алгоритмически в DRAM (как 5O-1).
      Раунд 1 (сделано): `OutByteCount` (вызов на каждый выходной байт) — убраны
      избыточные `PUSH/POP HL/DE/BC/AF` (вызывающие на сохранность не полагаются),
      байт держим в C. ~84T/байт. Сборка чистая. **Ждёт железа.**
- [~] **5O-6. Горячий вывод в SRAM.** Сделано: `Lh1PutByte` (вызов на каждый
      выходной байт) оптимизирован (байт в C без PUSH/POP AF; `text_buf[r]` пишется
      по адресу r напрямую, т.к. TextBufBase=#0000) и перенесён в SRAM-бандл.
      Flush сбрасывает `Lh1OutPos` уже при CASH_ON.
      Раунд 2 (сделано): основной цикл вынесен в SRAM (`CacheLh1Loop`) +
      `CacheRemainingZero`; DecodeLh1 (DRAM) делает setup/Enter/init, зовёт
      `CacheLh1Loop`, затем финальный flush/Restore. match-copy оптимизирован
      (text_buf[matchI] по адресу=index, без перезагрузки Lh1MatchI). Теперь ВЕСЬ
      горячий путь lh1 в SRAM; DRAM только на DSS-границах (трамплины). lh5
      по-прежнему зовёт DRAM `RemainingZero`. Сборка чистая. **Ждёт железа.**
- [ ] **5O-5. Крупнее буферы ввода/вывода** — реже DSS-трамплины (мелочь относительно
      поцикловой работы; делать в последнюю очередь).
**Замечание:** на `-lh1-` догнать unzip нереально (формат тяжелее DEFLATE), но
разрыв реально сократить. **Проверка каждого шага:** CRC16 = OK на всех `test/`
+ замер времени до/после на одних и тех же файлах.

### Заметки по проверке производительности (этап 5)
Замерять одинаково до/после: один и тот же набор файлов, turbo-режим,
фиксировать время распаковки (например, по таймеру/счётчику кадров) и
сравнивать DRAM-версию (этапы 3–4) с SRAM-версией.
