# CLAUDE.md — UNLHA: распаковщик LZH/LHA для Sprinter DSS

Этот файл описывает требования к проекту и конвенции, которым нужно следовать.
Источник истины по изменениям — этот репозиторий. Соседние проекты используются

## 1. Что это

`UNLHA` — консольная утилита для Sprinter DSS, которая распаковывает архивы
формата **LZH/LHA** (https://ru.wikipedia.org/wiki/LHA). Утилита пишется на
**ассемблере Z80** с упором на **максимальную скорость распаковки** за счёт
использования особенностей архитектуры Sprinter — в первую очередь исполнения
«горячего» кода из **SRAM CACHE** (где код не тормозится wait-состояниями).

Аналоги, на которые ориентируемся (одна платформа, тот же приём с кэшем):
- `/Users/dmitry/dev/zx/sprinter/sources/sprinter-unzip` — распаковщик ZIP/DEFLATE,
  SRAM-кэш для inflate-петли и CRC, формат DSS EXE, сборка, образ дискеты.
- `/Users/dmitry/dev/zx/sprinter/gifview` — LZW-декодер в SRAM-кэше, Makefile +
  `tools/build.sh`, аккуратная организация `cache_code.asm` через `DISP`.

## 2. Формат LZH/LHA (что должны разбирать)

Архив = последовательность записей. Каждая запись: заголовок + сжатые данные.
Заголовки бывают уровней 0/1/2/3; в наших тестах встречаются уровни 0 и 1.

Заголовок Level 0:
```
off  size  поле
00h  1     Header Size (= 16h + F + E)
01h  1     Header Checksum (сумма байт [02h..конец заголовка] & 0xFF)
02h  5     Метод сжатия, ASCII, например "-lh5-"
07h  4     Compressed Size   (LE)
0Bh  4     Uncompressed Size (LE)
0Fh  2     MS-DOS time
11h  2     MS-DOS date
13h  1     атрибут (обычно 20h)
14h  1     Header Level (00h)
15h  1     длина имени F
16h  F     имя файла
16h+F 2    CRC16 распакованных данных (init 0000h)
```
Level 1 похож, добавляет OS ID и цепочку Extended Headers (поле «размер
следующего расширенного заголовка», 0 = конец). Level 2/3 — двухбайтовый размер
заголовка, CRC16 на фиксированном смещении, дата как unix-time. Подробная
раскладка — https://problemkaputt.de/psxspx-cdrom-file-compression-larc-lharc-lha-lzs-lzh.htm
и http://fileformats.archiveteam.org/wiki/LHA

**Контроль целостности — CRC16** (полином 0xA001 / «ARC», init 0x0000), это
по распакованным данным. Это отличие от ZIP (там CRC32) — не перепутать.

### Методы сжатия

| Метод  | Длина  | Окно   | Алгоритм                         | Приоритет |
|--------|--------|--------|----------------------------------|-----------|
| -lh0-  | —      | —      | хранение без сжатия              | MVP       |
| -lhd-  | —      | —      | запись-каталог (нет данных)      | MVP (skip)|
| -lh5-  | 3..256 | 8 KB   | LZSS + статич. Хаффман (AR002)   | **MVP**   |
| -lh1-  | 3..60  | 4 KB   | LZHUF (LZSS + адаптив. Хаффман)  | **MVP**   |
| -lh4-  | 3..256 | 4 KB   | AR002, окно 4 KB                 | после MVP |
| -lh6-  | 3..256 | 32 KB  | AR002, окно 32 KB                | после MVP |
| -lh7-  | 3..256 | 64 KB  | AR002, окно 64 KB                | после MVP |
| -lzs-/-lz5- | 2..17 | 2/4 KB | LZSS (LArc)                  | опционально |

В тестовых файлах (каталог `test/`) реально встречаются:
- `clock.lzh`, `wios110r.lzh` → **-lh5-**
- `CPM431.lzh`, `GS19.LZH`, `WINPROFI.LZH` → **-lh1-**

Поэтому **MVP обязан поддерживать `-lh0-`, `-lh5-` и `-lh1-`**. Кодек AR002
общий для -lh4/5/6/7-, они различаются только размером окна — после рабочего
-lh5- добавление остальных AR002-методов малозатратно.

#### -lh5- (AR002), основной формат
Поток разбит на блоки. В начале блока — число символов в блоке (16 бит), затем
три таблицы кодов Хаффмана, заданные длинами кодов:
1. таблица длин кодов для «таблицы P» (кодирует сами длины, дельта-кодирование);
2. главная таблица: литералы 0..255 + коды длины совпадения (256..) — её длины
   кодируются через таблицу из п.1;
3. таблица расстояний (старшие биты позиции; младшие читаются битами явно).
Декодирование символа: литерал → байт в окно; иначе код длины (3..256) и код
расстояния (LZSS-ссылка в скользящем окне). Биты читаются MSB-first.

#### -lh1- (LZHUF), второй формат
Адаптивный (динамический) Хаффман для литералов/длин: дерево перестраивается
по ходу декодирования (инкремент частот, ребалансировка). Расстояние кодируется
6-битным префиксом через статическую таблицу + явные младшие биты. Окно 4 KB,
длина совпадения 3..60. Алгоритм — классический `LZHUF.C` (Okumura/Yoshizaki).

## 3. Платформа Sprinter — что нужно знать

Документация: `/Users/dmitry/dev/zx/sprinter/sprinter_ai_doc/manual`.

### Окна памяти (16 KB каждое) и порты страниц
```
WIN0  #0000-#3FFF   порт PAGE0 = #82
WIN1  #4000-#7FFF   порт PAGE1 = #A2
WIN2  #8000-#BFFF   порт PAGE2 = #C2
WIN3  #C000-#FFFF   порт PAGE3 = #E2
```
`OUT (PAGEx),A` — отобразить физическую страницу A в окно. `IN A,(PAGEx)` —
прочитать текущую страницу окна (для сохранения/восстановления).

### Формат DSS EXE (повторяем как в sprinter-unzip)
512-байтный заголовок по `ORG #4000`, код по `ORG #4200`:
```asm
        ORG     0x4000
ExeHeader:
        DB      "EXE"
        DB      ExeVersion          ; = 1
        DW      0x0200              ; смещение кода в файле (= размер заголовка)
        DW      0                   ; hi-слово 32-битного смещения
        DW      0                   ; primary loader size
        DW      0,0,0               ; reserved
        DW      Start               ; адрес загрузки  (#4200)
        DW      Start               ; точка входа     (#4200)
        DW      StackTop            ; адрес стека     (#7FFE)
        DS      512 - 22, 0
StackTop EQU    0x7FFE
        ORG     0x4200
Start:  ...
```
Сборка: `sjasmplus --raw` даёт сырой бинарь, который копируется в `*.EXE`
(заголовок «EXE» — внутри бинаря, как выше). DSS-загрузчик читает магию/версию,
грузит код по адресу загрузки и ставит SP.

### Вызовы DSS и BIOS
- **DSS**: `RST #10`, номер функции в `C`, ошибка — флаг **CF=1**.
- **BIOS**: `RST #08` (используется EMM FN4: blockID+страница → физ. страница).

Нужные функции DSS (номера в `C`):
```
#11 Open        HL=имя, A=режим            -> A=handle (CF=err)
#12 Close       A=handle
#13 Read        A=handle, HL=буфер, DE=кол -> DE=прочитано, A=0/FF, CF=err
#14 Write       A=handle, HL=буфер, DE=кол -> DE=записано, CF=err
#0B Create      HL=имя, A=атрибут          -> A=handle (CF=err)
#15 Move_FP     A=handle, DE=смещ, B=режим (set/cur/end)
#3D GetMem      B=число страниц 16K        -> A=blockID (CF=err)
#3E FreeMem     блок по ID
#30 WaitKey                                -> A=код
#5C PChars      HL=ASCIIZ-строка
#41 Exit        A=код возврата
```
Получение физической страницы выделенного блока (BIOS EMM FN4):
```asm
        LD      A,(BufferBlock)     ; blockID из GetMem
        LD      B,0                 ; индекс страницы внутри блока
        LD      C,BiosEmmFn4
        RST     0x08                ; -> A = физ. страница
        OUT     (PAGE2),A           ; отобразить в окно
```
Точные номера/нюансы сверять с `sprinter-unzip`, `gifview/include/*.inc` и
manual'ом перед использованием.

### SRAM CACHE — ключевой приём ускорения

Идея: «горячий» код (петля распаковки, чтение битов, обход дерева Хаффмана,
копирование совпадений, обновление CRC) и часто читаемые таблицы кладутся в
SRAM окна 0, где **нет wait-состояний** (в turbo 21 МГц). Холодный код (UI,
парсинг заголовков, диалоги) остаётся в обычной странице EXE.

Механика (как в обоих референсах):
1. Горячий код пишем в отдельном участке исходника под `DISP <SRAM-адрес>` —
   sjasmplus ассемблирует метки под адреса SRAM, но байты физически лежат в EXE.
2. На старте копируем этот блок (`LDIR`) в SRAM и выполняем оттуда.
3. Вход/выход в режим кэша — парой подпрограмм (имена как в референсах):
```asm
EnterCacheWindow:           ; включить SRAM-окно, запретить прерывания
        PUSH    BC
        XOR     A
        LD      BC,0x1FFD
        OUT     (C),A               ; ISA system <- 0
        LD      A,0x04
        OUT     (SysPortOff),A      ; system map -> режим SRAM (#3C)
        DI
        IN      A,(0xFB)            ; CASH_ON = 1
        POP     BC
        RET
RestoreSystemWindow:        ; вернуть обычное отображение, разрешить прерывания
        PUSH    BC
        IN      A,(0x7B)            ; CASH_ON = 0
        LD      A,0x03
        OUT     (SysPortOff),A      ; system map -> режим DSS
        LD      BC,0x1FFD
        LD      A,0x01
        OUT     (C),A               ; ISA system <- 1
        EI
        POP     BC
        RET
```
4. Любой вызов DSS (чтение/запись файла) **нельзя** делать с включённым кэшем:
   на границе ввода-вывода временно `RestoreSystemWindow` → DSS → `EnterCacheWindow`
   (паттерн `LOAD_NXT_BLOCK_CACHE`/`WRITE_BUFF_CACHE` из sprinter-unzip).
5. Точные значения портов/режимов (`#3C`, `#7C`, `#FB`, `#7B`, `#1FFD`,
   `CASH_ON`) брать из manual + референсов, не из памяти.

Что класть в SRAM для UNLHA:
- петлю декодера (-lh5-/-lh1-), чтение битового потока, обход Хаффмана;
- копирование совпадений из скользящего окна;
- таблицу/процедуру обновления CRC16;
- таблицы Хаффмана текущего блока (для -lh5-) / дерево (для -lh1-).

## 4. Раскладка памяти (план, уточняется при реализации)

```
WIN0 #0000-#3FFF  SRAM CACHE: горячий код + таблицы Хаффмана + CRC16 + BSS декодера
WIN1 #4000-#7FFF  EXE: заголовок(#4000), холодный код+данные(#4200), стек(#7FFE)
WIN2 #8000-#BFFF  скользящее окно LZSS / выходной буфер (страница из GetMem)
WIN3 #C000-#FFFF  выходной/входной буфер (страница из GetMem)
```
Для -lh5- окно 8 KB помещается легко; для -lh6/-lh7- (32/64 KB) понадобится
многостраничный кольцевой буфер через перелистывание WIN2/WIN3 — учитывать в
дизайне, но не в MVP.

## 5. Сборка

- Ассемблер: **sjasmplus**. Сборка через **Makefile** (как в `gifview`:
  тонкий Makefile + `tools/build.sh`).
- Цели Makefile (ожидаемые):
  - `make` / `make exe` → `build/UNLHA.EXE` (через `sjasmplus --raw`, затем копия в `.EXE`);
  - `make img` → `build/unlha.img` — образ дискеты для **MAME** с `UNLHA.EXE` и
    тест-архивами (через `mtools`: `mformat -f 1440`, `mcopy`; либо `fformat`,
    см. `/Users/dmitry/dev/zx/sprinter/sources/fformat`);
  - `make run` → запуск MAME с образом (если настроено);
  - `make clean`.
- Артефакты — в `build/` (не коммитить бинарники без необходимости).
- Полезно генерировать `--lst`/`--sym`/`--sld` для отладки (DeZog/MAME).

- Желательны автоматические проверки: распаковать, сверить CRC16/размер, при
  возможности сравнить с эталонным выводом.

## 7. Кодстайл

- Стиль — как в приложенных проектах (`gifview`, `sprinter-unzip`): метки в
  колонке 0, инструкции с табуляцией, выровненные комментарии после `;`.
- **Нейминг подпрограмм и переменных — CamelCase** (как в `gifview`:
  `CacheDecodeCurrentFrameToCanvas`, `FrameStreamPage`). Префикс `Cache…` —
  для кода/данных, исполняемых из SRAM-кэша.
- Константы/equ и порты — допустимо КАПСОМ или CamelCase согласованно в файле
  (порты: `PAGE0`, `PAGE1`…; функции DSS: `DssOpen`, `DssRead`… в CamelCase).
- Числа — в hex как `0x…` или `…h`, единообразно в пределах файла.
- Горячий SRAM-код — в отдельном файле (по образцу `gifview/src/cache_code.asm`),
  оформлять через `DISP`, выравнивать важные таблицы по границам.
- Комментарии у нетривиальных мест: вход/выход регистров, что портится.
- Инварианты раскладки памяти страховать `ASSERT` (как в `sprinter-unzip`).

## 9. Интерфейс командной строки (как в sprinter-unzip)

CLI повторяет модель `sprinter-unzip` (`unzip.asm`, `README.md`): один и тот же
синтаксис флагов, масок, выходного каталога и файла листинга.

Синтаксис:
```
unlha.exe <archive.lzh> [<out_dir>] [<mask>]            ; распаковать
unlha.exe {-l|/l} <archive.lzh> [<list_file>] [<mask>]  ; список
unlha.exe {-o|/o} <archive.lzh> [<out_dir>] [<mask>]    ; перезаписывать
unlha.exe {-s|/s} <archive.lzh> [<out_dir>] [<mask>]    ; пропускать существующие
unlha.exe {-x|/x} <archive.lzh> [<out_dir>] <mask>      ; срезать префикс каталога маски
```
Режимы/флаги (префикс `-` или `/`, односимвольные):
- `-l` — только вывести содержимое архива (список);
- `-o` — перезаписывать существующие файлы;
- `-s` — пропускать существующие файлы;
- `-x` — срезать с имён префикс каталога из маски (нужна маска с разделителем).

**Список (`-l`)**: при отсутствии `<list_file>` печатается на экран с
постраничной паузой (как в sprinter-unzip — `LIST_PAGE_LINES = 30`). Если задан
`<list_file>` (имя без wildcard) — листинг пишется в текстовый файл
(`DssCreate`/`DssWrite`), при необходимости создаются каталоги пути. Формат
строки: дата/время, размер, имя (и метод сжатия).

**Каталог распаковки**: по умолчанию — **текущий** (`GetCurDir`). Можно задать
позиционным `<out_dir>` (приводится к абсолютному пути, добавляется завершающий
`\`). Если второй аргумент содержит wildcard — он трактуется как маска, а вывод
идёт в текущий каталог. Отдельного флага `-d` нет — каталог позиционный.

**Маски/фильтры файлов**: `<mask>` отбирает записи по имени.
- `*` — любая последовательность, `?` — один символ;
- сравнение регистронезависимое; `/` и `\` эквивалентны;
- маска, оканчивающаяся на `\` или `/`, означает «всё в этом каталоге»
  (неявно добавляется `*`): `data\` ≡ `data\*`.
Реализация по образцу `MATCH_ENTRY_MASK` / `NORMALIZE_MASK_TRAILING` /
`APPLY_STRIP_PREFIX` из sprinter-unzip.

**Чтение командной строки DSS**: хвост команды доступен через `IX` (`IX+0` —
длина, далее строка), токены разделяются пробелами/управляющими (< 0x20).
Парсинг — по образцу `GetCmdParam`: пропуск пробелов, копирование до
разделителя, null-терминация, `CF=1` если параметров больше нет.

**Коды возврата** (как в sprinter-unzip, передаются `DssExit`):
- `0` — успех; `1` — неверная командная строка либо прерывание (`Esc`/`Ctrl+C`);
- `3` — архив не найден; `7` — выходной файл существует и не заменён;
- `13` (0x0D) — повреждён/неизвестен формат LZH; `23` (0x17) — несовпадение CRC;
- прочее — проброс кода ошибки DSS. Сохраняется первая ненулевая ошибка.
Прерывание (`Esc`/`Ctrl+C`) опрашивать в петлях списка и распаковки.
